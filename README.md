# AI Cloud Architecture Generator

A SaaS tool where a user describes a product idea in plain language and the system takes it
through progressive, LLM-assisted cloud architecture design: a conversational brainstorm to pull
out functional and non-functional requirements, an abstract high-level design generated by a
deterministic rules engine, multi-cloud (AWS / Azure / GCP / Kubernetes / private cloud) service
mapping with cost estimates, low-level per-provider configuration detail, and Terraform /
Kubernetes-manifest export. As a project's requirements grow, the system produces incremental
"what changed" delta updates instead of regenerating from scratch, and supports manual HLD edits
as an alternative to LLM-driven regeneration.

Every architectural decision — at every level of detail — carries a stored, non-optional
rationale (e.g. `HldContent.components[].reasoning` cannot be empty in the schema). The rules
engines are deterministic and run first; the LLM's job is to validate/adjust and narrate those
decisions, not invent architecture from scratch.

## Architecture at a glance

```
Browser ⟷ Next.js (public gateway) ⟷ [private network] ⟷ FastAPI (never public) ⟷ Postgres (pgvector) / Redis
```

- **Next.js** (App Router, Turbopack, repo root) is a thin gateway only. Its single catch-all
  route (`src/app/api/[...path]/route.ts`) forwards every `/api/*` request to the backend,
  strips hop-by-hop headers, attaches a shared `x-internal-auth` secret, forwards the real
  client IP, and streams the response through. No other frontend code talks to the backend or a
  database directly.
- **FastAPI** (`backend/`) owns Postgres, Redis, and all business logic. It has no CORS
  middleware by design — it's only ever called from Next.js's own server-side proxy, never a
  browser origin — and enforces the `x-internal-auth` header on every request as defense-in-depth
  on top of network isolation.
- **Postgres** (via the `pgvector/pgvector:pg16` image, for the knowledge-base similarity search
  feature) and **Redis** back the FastAPI service.

See `HANDOFF-backend-split.md` for the history of the split from a single Next.js monolith into
these two services.

## Prerequisites

- Node.js 22+
- Python 3.12+
- Docker (with Docker Compose) for Postgres, Redis, and the backend

## Local development setup

1. Copy the example env files and fill in real values:
   ```bash
   cp .env.example .env
   cp backend/.env.example backend/.env
   ```
   `INTERNAL_AUTH_SECRET` must match **exactly** between the two files — a mismatch fails every
   request with a 401.

2. Start Postgres, Redis, and the backend in Docker:
   ```bash
   docker compose up -d postgres redis backend
   ```

3. Install frontend dependencies and run the dev server on the host:
   ```bash
   npm install
   npm run dev
   ```

4. Confirm the wiring is alive:
   ```bash
   curl http://localhost:3000/api/health
   # {"ok":true}
   ```

The frontend also has its own containerized production-build path (`docker compose up frontend`,
port 3001) for testing the real production build end-to-end — this is not the day-to-day dev
workflow, which stays `npm run dev` against the Dockerized backend on port 8000.

## Running tests

- **Frontend**: no test suite exists yet (`package.json` has no `test` script). Noting this
  explicitly rather than implying coverage that doesn't exist.
- **Backend**: `pytest`, already configured, one test file (`backend/tests/test_llm.py`) covering
  the LLM fallback-chain service; all HTTP calls are mocked (`respx`), no real database needed to
  run it.
  ```bash
  cd backend
  python3.12 -m venv .venv && .venv/bin/pip install -e ".[dev]"
  .venv/bin/python -m pytest -q
  ```

## Running migrations

```bash
cd backend
alembic upgrade head
```

## CI

`.github/workflows/ci.yml` runs on every push/PR to `main`: frontend typecheck/lint/build plus
`npm audit --audit-level=high`, and backend `pytest` plus `pip-audit`, both required to pass.
`.github/dependabot.yml` checks weekly for npm, pip, and GitHub Actions updates.

## Deployment

There is no cloud-specific Infrastructure-as-Code for hosting this app yet, and a target
platform/orchestrator has not been chosen. What's already in place: both containers
(`backend/Dockerfile`, root `Dockerfile`) run as non-root users and ship a `HEALTHCHECK`, and
`docker-compose.yml` wires matching `healthcheck:` blocks for the `backend` and `frontend`
services (see the CI/container-hardening changes in this same PR). Choosing and wiring up an
actual deploy target (ECS/Fargate, Kubernetes, Fly.io, Render, etc.) is still open work.
