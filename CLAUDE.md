## Product Vision

A SaaS tool where a user describes any product idea in plain language. The system:
1. Brainstorms conversationally to understand requirements (multi-turn LLM, not one-shot Q&A).
2. Extracts structured functional + non-functional requirements.
3. Generates architecture in progressive detail: **abstract HLD → multi-cloud (AWS/Azure/GCP) service mapping → LLD (config-level detail) → Terraform export**.
4. As the product grows, generates **incremental delta updates** ("What Changed" diff + cost delta) instead of regenerating from scratch, and supports **manual HLD edits** as an alternative to LLM-driven regeneration.

**Core differentiator**: every architectural decision, at every level of detail, carries a stored, genuine rationale. This is enforced structurally, not just by prompting:
- `HldContent.components[].reasoning` is a required (non-optional) string field in the DB schema — a component literally cannot be persisted without one.
- Cloud mappings carry `alternatives: [{ serviceName, reason }]` — rejected options and why, not just the winner.
- LLD config values carry a parallel `reasoning: Record<string, string>` map, one rationale string per config key.
- The rules engines (`rules-engine.ts`, `lld-rules.ts`, `cloud-mapping.ts`) are deterministic and run first; the LLM's job (`llm.ts`) is to validate/adjust and *narrate* those decisions, not invent architecture from scratch. This is a real architectural boundary, not just a description — see "Reasoning Core" below.

## Backend/Frontend Split

The app was split from a single Next.js monolith into two genuinely separate services for real network isolation (this is meant to be a production/enterprise app, not a demo):

```
Browser ⟷ Next.js (public, port 3000) ⟷ [private network] ⟷ FastAPI (never public, port 8000) ⟷ Postgres (5433) / Redis (6379)
```

- **Next.js is a thin gateway only.** The single catch-all route `src/app/api/[...path]/route.ts` forwards every `/api/*` request to the backend — strips hop-by-hop headers, attaches the shared `X-Internal-Auth` secret, forwards the real client IP (not a spoofable client-supplied one), streams the response body, 60s timeout. This file is the *only* place that reads `BACKEND_URL` / `INTERNAL_AUTH_SECRET`. No other frontend code talks to the backend or a database directly anymore.
- **FastAPI (`backend/app/main.py`) has no CORS** (by design — it's only ever called from Next.js's own server-side proxy, never a browser origin) and enforces the `X-Internal-Auth` header on every request as defense-in-depth on top of network isolation. All errors are normalized to `{"error": string}` to match what the frontend already expects.
- Auth/login is **explicitly deferred** — no users table, no sessions. Was an explicit user decision to get the app working end-to-end first before adding login.

### Running locally

```bash
docker compose up -d        # postgres (5433), redis (6379), backend (8000) — all in a private `internal` docker network
npm run dev                 # Next.js dev server, runs bare-metal on the host (not containerized), port 3000
```

- Two ports to keep straight: frontend `:3000` (what you open in the browser), backend `:8000` (only reachable from the host/Next.js locally; never published in a real deployment).
- `INTERNAL_AUTH_SECRET` must be set **identically** in both root `.env` (read by the Next.js proxy) and `backend/.env` (read by FastAPI's middleware) — a mismatch fails every request with `401 Unauthorized`. `BACKEND_URL` (root `.env`) should point at `http://localhost:8000` for local dev.
- `backend/.env` also carries `DATABASE_URL`, `OPENROUTER_API_KEY`, and `REDIS_URL` — these no longer live in the frontend's `.env` at all.
- Checkpoint to confirm the wiring is alive: `curl http://localhost:3000/api/health` → `{"ok":true}`, and a request without `X-Internal-Auth` hitting the backend directly on `:8000` should 401.

### Phase 3 Step 2 (interactive manual HLD editor) — **PARTIALLY BUILT, not "zero code" as previously assumed**

This is the most important correction from this audit. `ArchitectureWorkspace.tsx` already contains a substantially working manual editor, and there is a dedicated, functioning backend endpoint for it:

**Already implemented:**
- Edit-mode toggle (`isEditing`/`draftHld` state, deep-cloned from the live architecture so edits are a draft until saved)
- Add component (form: type + name + optional reasoning) and remove component (× button on each SVG node)
- Add connection (from/to/protocol form) and remove connection (× chip in the active-connections list)
- Edit a component's top-level `reasoning` text (textarea in the drawer, edit mode only)
- Edit LLD config values and per-key LLD reasoning strings, per cloud provider, for the currently selected component (drawer accordion, edit mode only)
- **Live client-side validation** while editing: `validateArchitectureLayout()` (`src/lib/validation.ts`) runs reactively on every `draftHld`/`activeProvider` change, surfacing hard errors (orphaned components, compute-without-database) as blockers and soft warnings (CDN/LB wired directly to DB, cost exceeding stated budget) — save is disabled while `errors.length > 0`
- Save flow: `POST /api/projects/[id]/architectures/manual` (`src/app/api/projects/[id]/architectures/manual/route.ts`) — auto-fills cloud mappings/LLD baselines for any newly-added component that lacks them, re-runs `validateArchitectureLayout` server-side as a gate, computes the added/removed/modified diff **in TypeScript** (not via LLM, unlike the auto-generate path) by comparing component IDs and LLD config keys against the previous version, and inserts a new versioned `architectures` row exactly like the auto-generate path does
- Manually-touched components are tagged `metadata: { isManuallyAdded, overrideSource: "user" }` and rendered with a dashed border + amber "U" badge in the SVG diagram to distinguish them from rule-engine output

**Genuinely not implemented (real remaining scope for this step):**
- No way to change which cloud *service* a component is bound to (e.g. swap AWS Lambda → Fargate) — only LLD config values under the already-assigned service are editable
- No drag-to-reposition on the canvas — node position is still fully computed by the fixed column-bucketing layout algorithm (`getLayout()`), so manually added components just fall into a column by `type`
- No cycle detection or connection-count limits beyond the orphan/dependency checks already in `validation.ts`
- No undo/redo within a single edit session (only "Cancel" which discards the whole draft)

**Recommendation before writing new code for this step**: read `ArchitectureWorkspace.tsx` in full and decide whether the remaining scope above is what "Step 2" is meant to cover, or whether the intent was a different UI altogether (e.g., a draggable canvas). Don't re-build what's already there.

## Versioning Pattern (applies throughout)

`projects` (1 row per project, mutable `currentVersion` pointer) →
`conversations` (append-only, tagged with `stage`, never updated) →
`requirements` (append-only, integer `version`, `PUT`/extraction always inserts a new row — never updates in place) →
`architectures` (append-only, semver-string `version` with the patch segment bumped on every generate/manual-save, each row is a **full self-contained snapshot**: complete `hld` (components + connections + all three providers' cloud mappings + LLD) and `reasoning` (decisions, assumptions, risks, recommendation, diff) — nothing is normalized out into separate decision/mapping tables, despite the original planning scaffold (`planning-data.ts`) describing a more normalized schema)

Both `architectures` insert paths (LLM auto-generate in `architectures/route.ts`, and manual edit in `architectures/manual/route.ts`) independently compute a `diff` block against the immediately-previous version by `createdAt desc`, and both bump `projects.currentVersion` after insert.
